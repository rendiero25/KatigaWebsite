---

## Decomposisi Subsistem

### Phase 1 — Payment Lifecycle + Biteship Tracking `[PRIORITAS UTAMA]`

**A. Payment Lifecycle (pending timer, close Snap, expiry)**
- Timer 24 jam di halaman detail pesanan ketika status `awaiting_payment`
- Ketika user close Snap popup atau tutup tab → redirect ke `/pesanan/:id` dengan status pending (sudah ada onClose, perlu dipercantik)
- `expire` webhook dari Midtrans → update `orderStatus: 'cancelled'`, `paymentStatus: 'expired'`
- Tampilkan countdown waktu sisa bayar di UI

**B. Biteship Tracking di Order Detail**
- Tampilkan `biteshipTrackingCode` / resi secara visual di halaman pesanan
- Tombol "Cek Status Pengiriman" → panggil Biteship tracking API
- Status kurir real-time (pickup, in-transit, delivered)
- Update `orderStatus: 'delivered'` otomatis via Biteship webhook atau polling

---

### Phase 2 — Cancellation System

- Customer cancel order **sebelum** admin packing (status `awaiting_payment` atau `processing` sebelum di-accept admin)
- Admin toggle: "Pesanan diterima / mulai packing" → setelah ini customer tidak bisa cancel
- Jika cancel saat `paymentStatus: 'pending'` → Midtrans cancel/expire transaction
- Jika cancel saat `paymentStatus: 'paid'` → Midtrans refund API → `paymentStatus: 'refunded'`
- Guard di backend: blok cancel jika admin sudah set status ke packing/shipped

---

### Phase 3 — Invoice PDF

- Invoice sukses: generate setelah pembayaran berhasil (dapat didownload dari halaman pesanan)
- Invoice batal: generate setelah order dibatalkan (konfirmasi pembatalan + refund jika ada)
- Library kandidat: `pdfkit` atau `@react-pdf/renderer` (client-side), atau endpoint backend generate PDF
- Kirim via email (Resend) sebagai attachment opsional

---

### Phase 4 — Komplain & Retur Barang

- Model baru: `Complaint` (orderId, customerId, type: komplain|retur, alasan, foto bukti, status: open|processing|resolved|rejected)
- Flow: customer buka komplain dari halaman detail pesanan (hanya jika `orderStatus: 'delivered'`)
- Admin: lihat semua komplain, bisa approve/reject retur
- Jika retur diapprove: trigger Midtrans refund
- Batas waktu komplain: misal 7 hari setelah `delivered`

## Hal Tambahan yang Perlu Dipastikan

| Item | Keterangan |
|------|-----------|
| Stock decrement | Kurangi stok produk saat order confirmed (paid), bukan saat cart. Sudah ada `soldCount` di webhook, tapi belum ada stok tracking. |
| Payment expiry otomatis | Midtrans kirim webhook `expire` setelah 24 jam → order otomatis `cancelled` + `expired` |
| Admin order management | Accept/reject cancellation, mark packing, mark shipped — perlu flow di admin panel |
| Email transaksional | Konfirmasi order, pembayaran berhasil, pesanan dikirim, order dibatalkan (via Resend) |
| Rate limiting | Endpoint cancel/komplain perlu dibatasi agar tidak disalahgunakan |
| Notifikasi real-time | Push notif ke customer ketika status order berubah (sudah ada sistem notifikasi) |
| Guard cancellation | Validasi di backend (bukan hanya frontend) agar cancel hanya bisa saat status belum packing |

Phase 1 jadi fondasi — Phase 2, 3, 4 depend pada lifecycle order yang benar.
